SWT-Bench is a benchmark for software testing capabilities on real world software issues.

SWT-Bench is based on real-world GitHub repositories of popular python libraries. Given a code base and a user-reported issue, the task is to generate a reproducing test, which fails in the original state of the code base and passes after a patch resolving the issue has been applied.

# What is the task in SWT-Bench?

Models should be able to generate software tests which reproduce user-reported issues and increase code coverage.

SWT-Bench is based on popular Python repositories scraped from GitHub. Each task is based on a Pull-Request in the repository which resolves a reported user issue by applying a code patch and adding unit tests to reproduce the reported issue. Given the code base before the patch resolution and the user issue, models are instructed to generate tests that reproduce a reported user issue. Generated tests are expected to fail on original code bases for the reason described in this issue and pass after it is resolved.

# What do these numbers mean?

We measure how precisely generated unit tests capture user issues and whether they increase code coverage.

Models are instructed to generate tests that reproduce a reported user issue. Generated tests are expected to fail on original code bases for the reason described in this issue and pass after it is resolved. We denote the rate of creating such Fail-to-Pass tests as success rate \( \mathcal{S}\). The test should also cover lines relevant to the issue, specificially lines changed when resolving the issue, which is denoted as Coverage Increase \(\Delta \mathcal{C}\).

# What were the initial findings of SWT-Bench?

Code Agents are highly effective at writing unit tests, even when initially designed for code repair and not specifically adapted.

We evaluated the performance of bare bone LLMs in a Zero-Shot setting, the heuristic method LIBRO and various code agents on SWT-Bench. We found that, suprisingly, code agents are highly effective at writing unit tests and even outperformed the previous state of the art LIBRO.

# Why is it important to have powerful approaches to create unit tests?

Reproduction tests comprise large parts of test suites and are tedious to create. Moreover, unit tests are powerful tools to cross-verify proposed software patches.

When users report issues, it is important to capture the reported issue and be able to confirm that a proposed patch resolves the issue. This is typically done by creating a test that reproduces the issue and fails on the original code base and passes after the patch is applied. This is a tedious and error-prone process, which can be automated by code agents. Moreover, generated tests can be used to cross-verify proposed patches and ensure that they correctly resolve the reported issue.

# What is the relationship between SWT-Bench and SWE-bench?

SWT-Bench and SWE-bench measure complimentary skills. Given a repository and an issue, SWT-Bench measures whether models can reproduce the given issue, while SWE-bench measures whether a model can resolve it.

SWT-Bench and SWE-bench are based on the same GitHub repositories and user-reported issues. While SWT-Bench measures the ability of models to reproduce a given issue, SWE-bench measures the ability of models to resolve it. The skill of reproducing an issue is a complementary and often useful for sucessfully resolving it.
